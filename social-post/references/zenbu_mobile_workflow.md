# Zenbu 手機引導發文工作流

用於使用者透過手機叫 Codex「幫我發文 / 幫我想一篇 / 找內容庫搭配」時。目標是把發文從「直接生成」改成「編輯引導 → 內容媒合 → 多平台草稿 → 排程與追蹤」。

## 0. 啟動問題

一開始只問必要問題，不超過 3 個：

1. 今天有沒有明確主題？沒有也可以。
2. 有沒有要搭配的事件或素材：生活場景、客戶對話、開發進度、圖片、影片、截圖？
3. 想發到哪些平台：Facebook / Threads / IG？如果不確定，先建議平台組合。

若使用者已經提供主題，直接進策略判斷，不重問。

## 1. 策略判斷

輸出一段簡短診斷，先不要寫全文。

判斷項：

- **題材類型**：外部案例 / 生活事件 / 開發日誌 / 客戶問題 / 產品觀點 / 反直覺觀點 / 資源分享 / 社群互動。
- **候選公式**：從 `formulas.md` 選 1-3 個；長文可用下方 L 系列。
- **長度**：短文 100-250 字 / 中文 350-700 字 / 長文 800-1400 字。
- **平台**：
  - Facebook：長敘事、生活場景、反思、不要正文外部連結。
  - Threads：短句、立場、可口語一點，通常 60-180 字或短 thread。
  - IG：需要圖片/影片；文字改成 caption，先問素材是否可用。
- **素材分流**：同一主題不要一稿多投；FB 長、Threads 短、IG 圖文/影片 caption。

## 2. 長文模板 L 系列

原公式庫偏短中篇。需要 800-1400 字長文時，用 L 系列搭配 F 系列：

| 模板 | 使用時機 | 骨架 |
|---|---|---|
| L1 外部案例 → 我的行業觀點 | YouTube / 新聞 / 書摘 | 外部洞察不提來源 → 套到我的場景 → 行業反思 |
| L2 生活場景 → 商業洞察 | 接小孩 / 客戶對話 / 開發日常 | 生活場景 → 一句反直覺 → 方法論 |
| L3 客戶問題 → 方法論拆解 | 顧問、接案、B2B | 客戶常問 → 真正問題 → 3-5 個具體症狀 → 解法 |
| L4 我以前想錯了 | 建立信任 | 過去觀點 → 事件打臉 → 新判斷 |
| L5 無聊但值錢 | SOP、制度、流程、產品化 | 大家想追刺激 → 真正值錢的是無聊重複 → 具體例子 |
| L6 兩派都錯 | 爭議觀點 | A 派 / B 派 → 兩邊盲點 → 第三條路 |
| L7 從救火到系統 | 勞資、顧問、營運 | 日常救火 → 不是單一事件，是缺系統 → 制度化 |
| L8 把能力變成產品 | Zenbu / 接案者 / AI 產品化 | 個人能力 → 可重複流程 → AI 產品 |
| L9 一天中的一個場景 | 降低說教感 | 小場景 → 對照外部觀點 → 收束 |
| L10 做到一半才懂 | 開發日誌 | 正在做的事 → 卡點 → 新理解 → 下一步 |

## 3. 內容庫媒合

預設內容庫：

- Obsidian: `/Users/luke/Documents/Obsidian Vault/YouTube Research`
- Alex Hormozi notes: `YouTube Research/Channels/Alex Hormozi`
- 本地機器資料：`/Users/luke/youtube-notion-worker/data`

媒合流程：

1. 用使用者主題抽關鍵詞，例如「接案者產品化」「勞資制度」「AI 內容」「信任」「無聊但值錢」。
2. 搜尋 Obsidian Markdown 標題與 transcript。
3. 找到至少 3 個候選方向，每個方向包含：
   - 方向名稱
   - 可搭配素材/影片 note
   - 適合公式
   - 建議平台與長度
   - 為什麼適合
4. 先請使用者選方向。**未選方向前不要寫完整文案**。

可用本地搜尋指令概念：

```bash
rg -n "關鍵詞1|關鍵詞2" "/Users/luke/Documents/Obsidian Vault/YouTube Research"
```

## 4. 文案撰寫與修正

使用者選方向後：

1. 讀對應 note / transcript 的必要片段。
2. 讀候選公式段與平台 reference。
3. 產出各平台版本：
   - Facebook：主文，通常中長。
   - Threads：濃縮成短句或 thread。
   - IG：caption + 圖片/影片搭配建議；若無素材，先問使用者要不要補。
4. 文案後附一段「策略說明」：用哪個公式、長度、CTA 強度。
5. 問使用者要改哪裡。修改到使用者滿意。

禁止：

- 未確認就發。
- FB / Threads 正文放外部連結。
- 把外部創作者名字硬塞進文中，除非使用者要求引用。
- 沒讀素材就假裝看過完整內容。

## 5. 排程與發布

草稿確認後再問：

- 要發哪些平台？
- 要立即發，還是排程？排程時間用使用者所在地時區。
- IG 是否已有圖片/影片？沒有就只做 caption 或請使用者補素材。

每次實際發布或排程前，必須取得使用者明確「確認」。

## 6. Notion 記錄

需要建立或更新 Notion 的發文工作台，建議欄位：

| 欄位 | 用途 |
|---|---|
| Title | 貼文題目 |
| Platform | FB / Threads / IG |
| Formula | F3 / F10 / L5 / L8... |
| Length | short / medium / long |
| Source Notes | Obsidian note path |
| Life Scene | 使用者提供的生活事件 |
| Draft | 最終草稿 |
| Status | Idea / Drafted / Scheduled / Published / Reviewed |
| Scheduled At | 排程時間 |
| Published URL | 發布連結 |
| Review At | 24h 回看時間 |
| Metrics | 讚、留言、分享、觸及、非追蹤者、儲存、私訊 |
| Learnings | 本篇學到的受眾偏好 |

若 Notion 資料庫尚未存在，先建議欄位並詢問是否建立。

## 7. 24 小時回看與受眾偏好庫

發布或排程後，詢問是否要 24 小時後回看。若使用者同意：

1. 建立回看提醒或 automation。
2. 回看時請使用者提供截圖或數字，或在可用工具允許時讀平台數據。
3. 回填 Notion。
4. 更新「Audience Preferences」資料庫。

Audience Preferences 建議欄位：

| 欄位 | 用途 |
|---|---|
| Audience Segment | 接案者 / 老闆 / 顧問 / AI 創作者 |
| Topic | 主題 |
| Formula | 使用公式 |
| Platform | 平台 |
| Signal | 留言多 / 分享多 / 儲存多 / 私訊多 / flop |
| Evidence | 對應貼文與數據 |
| Preference | 可重複的偏好判斷 |
| Avoid | 觀眾不吃的點 |

之後選題時，先查 Audience Preferences，再決定公式和角度。
